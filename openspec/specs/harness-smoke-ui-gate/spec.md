## Requirements
### Requirement: smoke:ui validation layer exists
The Validation Ladder in HARNESS.md SHALL include a new layer named `smoke:ui` that verifies the embedded UI assets serve correctly via HTTP.

#### Scenario: smoke:ui layer documented
- **WHEN** a developer reads HARNESS.md
- **THEN** the Validation Ladder section lists `smoke:ui` with its trigger conditions and invocation command
- **AND** the Change Types section documents which lanes require smoke:ui

#### Scenario: Lane × Change-Type matrix updated
- **WHEN** the change-type matrix is rendered in HARNESS.md
- **THEN** rows for `user-feature` and `bug-fix` include a column showing whether smoke:ui is required (yes if web/server-handlers touched)

### Requirement: smoke-ui.sh script
The repository SHALL include `scripts/smoke-ui.sh` that builds the dev binary, starts it on port 3199 with --serve-only --unsafe-no-auth, verifies the /ui/ HTML and all referenced asset URLs, and emits a deterministic PASS/FAIL log.

#### Scenario: Script runs end-to-end
- **WHEN** `./scripts/smoke-ui.sh` is invoked from the repo root
- **THEN** the script builds a binary at `/tmp/nano-brain-smoke/nano-brain`
- **AND** starts the binary on port 3199 in serve_only mode
- **AND** waits up to 15 seconds for `/health` to return 200 OK with `ready: true`
- **AND** fetches `/ui/` and asserts Content-Type starts with `text/html` and body contains `<script` tag
- **AND** for each JS asset referenced in `/ui/`: asserts HTTP 200, Content-Type `application/javascript`, body size > 1024 bytes
- **AND** for each CSS asset referenced in `/ui/`: asserts HTTP 200, Content-Type `text/css`, body size > 100 bytes
- **AND** tears down the server (kills PID)
- **AND** prints final line `=== smoke:ui PASS ===` on success
- **AND** exits with code 0 on PASS, non-zero on FAIL

#### Scenario: Asset MIME mismatch detected
- **WHEN** the server serves an `index-X.js` asset with Content-Type `text/html` (regression of #275)
- **THEN** smoke:ui FAILS with explicit message identifying the failing asset and observed Content-Type

#### Scenario: Asset body size too small detected
- **WHEN** the server serves an asset where body size is less than 1024 bytes (likely HTML fallback)
- **THEN** smoke:ui FAILS with the observed size

### Requirement: Pre-merge gate enforces smoke:ui evidence
The `phase_pre_merge` function in `scripts/harness-check.sh` SHALL check the PR diff for web-touching paths and require a fresh smoke:ui evidence file before marking pre-merge gate as PASS.

#### Scenario: Web-touching PR without evidence
- **WHEN** the PR diff includes a change under `web/src/`, `web/package.json`, `internal/server/handlers/`, `internal/server/webui/`, or `internal/server/routes.go`
- **AND** no `docs/evidence/*/smoke-ui-output.log` file exists OR none is newer than `scripts/smoke-ui.sh`
- **THEN** `harness-check.sh pre-merge` reports FAIL on check `3.8 smoke:ui evidence`

#### Scenario: Web-touching PR with stale evidence
- **WHEN** an evidence log exists but is older than `scripts/smoke-ui.sh`
- **THEN** the check reports FAIL (evidence may not reflect current script behavior)

#### Scenario: Web-touching PR with valid evidence
- **WHEN** the evidence log exists, is newer than the script, AND contains `smoke:ui PASS`
- **THEN** `harness-check.sh pre-merge` reports PASS on check `3.8`

#### Scenario: Non-web PR
- **WHEN** the PR diff includes no files matching the web-touching path patterns
- **THEN** `harness-check.sh pre-merge` reports SKIP on check `3.8 smoke:ui evidence`

### Requirement: Evidence format
Evidence files SHALL be plain text logs at `docs/evidence/<change-slug>/smoke-ui-output.log` containing chronological output of the smoke-ui.sh run.

#### Scenario: Evidence contains required markers
- **WHEN** an evidence file is generated by `smoke-ui.sh`
- **THEN** it contains a header `=== smoke:ui run YYYY-MM-DD ... ===`
- **AND** lists the binary path with file size
- **AND** lists each HTTP check performed with its result
- **AND** ends with `=== smoke:ui PASS ===` (success) or `=== smoke:ui FAIL: <reason> ===` (failure)
