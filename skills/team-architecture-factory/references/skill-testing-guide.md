## 1. Testing framework overview

Skill quality verification is a combination of **qualitative evaluation** and **quantitative evaluation**.


| Type | Method | Suitable for |
|------|--------|--------------|
| **Qualitative** | User directly reviews output | Subjective quality: writing style, design, creative works |
| **Quantitative** | Assertion-based automatic grading | Objectively verifiable: file creation, data extraction, code generation |

Core loop: **write → run test → evaluate → improve → re-test**

## 2. Test prompt authoring

### Principles

Test prompts must be **concrete and natural sentences that a real user would input**. Abstract or artificial prompts have low test value.


### Bad examples

```
"Process this PDF"
"Extract data"
"Generate a chart"
```

### Good examples

```
"From the 'Q4_sales_final_v2.xlsx' in my Downloads folder, add a profit
margin (%) column using column C (sales) and column D (cost). Then
sort by profit margin in descending order."
```

```
"Extract the 3-page table from this PDF and convert to CSV. The table
header is 2 rows — first row is category, second row is the actual
column name."
```

### Prompt diversity

- Mix **formal / casual** tones
- Mix **explicit / implicit** intent (direct file-format mention vs context-inferred)
- Mix **simple / complex** tasks
- Some include abbreviations, typos, casual expressions

### Coverage

Start with 2-3 prompts but design them to cover:
- 1 core use case
- 1 edge case
- (Optional) 1 composite task

## 3. Execution test: with-skill vs baseline

### 3-1. Comparative execution structure

For each test prompt, spawn two subagents **simultaneously** (in the same message, both with `run_in_background=true`):


**With-skill run:**
```
prompt: "{test prompt}"
skill path: {skill path}
output path: _workspace/iteration-N/eval-{id}/with_skill/outputs/
```

**Baseline run:**
```
prompt: "{test prompt}"  (same)
skill: none
output path: _workspace/iteration-N/eval-{id}/without_skill/outputs/
```

### 3-2. Baseline selection

| Situation | Baseline |
|-----------|----------|
| New skill creation | Run same prompt without skill |
| Existing skill improvement | Pre-modification skill version (snapshot preserved) |

### 3-3. Timing data capture

Save `total_tokens` and `duration_ms` **immediately** upon subagent-completion notification. This data is only accessible at the notification moment and cannot be recovered later.


```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

> **OpenCode note:** the completion notification for `run_in_background=true` tasks arrives via `<system-reminder>`. Capture timing data within the orchestrator's first turn after notification, before any other work.
>

## 4. Quantitative evaluation: assertion-based grading

### 4-1. Writing assertions

When the output is objectively verifiable, define assertions for automatic grading.

**Good assertions:**
- Objectively true/false determinable
- Descriptive name — clear what's being checked just from the result
- Validates the skill's core value

**Bad assertions:**
- Always pass regardless of skill presence (e.g., "output exists")
- Requires subjective judgment (e.g., "well written")

### 4-2. Programmable verification

If the assertion is code-verifiable, write it as a script. Faster, more reliable than visual check, reusable across iterations.


### 4-3. Non-discriminating assertion warning

Assertions that pass 100% in both configurations do NOT measure the skill's discriminating value. When you find one, either remove it or replace with a more challenging assertion.


### 4-4. Grading result schema

```json
{
  "expectations": [
    {
      "text": "profit margin column added",
      "passed": true,
      "evidence": "column 'profit_margin_pct' confirmed in column E"
    },
    {
      "text": "sorted by profit margin in descending order",
      "passed": false,
      "evidence": "original order preserved without sort"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}
```

## 5. Specialist agent utilization

Using specialist-role agents in test/evaluation improves quality.

### 5-1. Grader

Performs assertion-based grading, extracts verifiable claims from output, cross-validates.


**Role:**
- Per-assertion pass/fail judgment + evidence
- Extract factual claims from output and verify
- Feedback on eval quality itself (propose when assertions are too easy or vague)

### 5-2. Comparator (blind A/B)

Anonymizes two outputs as A/B and judges quality without knowing which used the skill.


**When to use:** when you want to rigorously verify "is the new version really better?" Can be omitted in general iterative improvement.

**Judgment criteria:**
- Content: accuracy, completeness
- Structure: organization, formatting, usability
- Overall score

### 5-3. Analyzer

Analyzes statistical patterns in benchmark data:
- Non-discriminating assertions (both configurations pass → no discrimination)
- High-variance evals (results vary widely per run → unstable)
- Time/token tradeoff (skill raises quality but also cost)

## 6. Iterative improvement loop

### 6-1. Feedback collection

Show output to user, collect feedback. Empty feedback is interpreted as "no issues".


### 6-2. Improvement principles

1. **Generalize feedback** — narrow fixes matching only the test example are overfit. Fix at the principle level.
2. **Remove non-weight-bearing content** — read transcripts; if the skill is making the agent do unproductive work, delete that part.
3. **Explain the why** — even if user feedback is terse, understand why it matters and reflect that in the skill.
4. **Bundle repeated work** — if the same helper script is generated in every test run, pre-bundle it in `scripts/`.

### 6-3. Iteration procedure

```
1. Modify skill
2. Re-run all test cases in new iteration-N+1/ directory
3. Show results to user (compared with previous iteration)
4. Collect feedback
5. Modify again → repeat
```

**Termination conditions:**
- User satisfied
- All feedback empty (no issues in any output)
- No more meaningful improvement possible

### 6-4. Draft → re-review pattern

When modifying a skill, write a draft, then **re-read with fresh eyes** and improve. Don't try to write perfectly in one pass — go through draft-review cycle.

## 7. Description trigger validation

### 7-1. Writing trigger-eval queries

Write 20 eval queries — 10 should-trigger + 10 should-NOT-trigger.

**Query quality criteria:**
- Concrete and natural sentences a real user would input
- Include specific details (file path, personal context, column name, company name)
- Mix length, tone, format
- Focus on **edge cases** rather than obvious answers


**Should-trigger queries (8-10):**
- Various expressions of same intent (formal / casual)
- Skill/file type not explicitly mentioned but clearly needed
- Non-mainstream use cases
- Competes with other skills but this skill should win

**Should-NOT-trigger queries (8-10):**
- **Near-miss is key** — similar keywords but other tools/skills fit better
- Obviously unrelated queries ("write fibonacci function") have no test value
- Adjacent domains, ambiguous expressions, keyword overlap but different context

### 7-2. Existing skill conflict check

Verify the new skill's `description` does not overlap with existing skills' trigger zones:

1. Collect descriptions of existing skill list
2. Verify should-trigger queries of the new skill don't mis-trigger existing skills
3. On conflict, describe boundary conditions more clearly in `description`


### 7-3. Automatic optimization (optional advanced)

When description optimization is needed:

1. Split 20 eval queries into Train (60%) / Test (40%)
2. Measure trigger accuracy with current description
3. Analyze failure cases to generate improved description
4. Select best description on Test set (NOT Train set — prevent overfit)
5. Up to 5 iterations

> This process is performed by automation script using `opencode run` CLI. Token cost is high, so run at the final stage after the skill is sufficiently stabilized.

## 8. Workspace structure

Directory structure for systematic management of test/evaluation results:

```
{skill-name}-workspace/
├── iteration-1/
│   ├── eval-descriptive-name-1/
│   │   ├── eval_metadata.json
│   │   ├── with_skill/
│   │   │   ├── outputs/
│   │   │   ├── timing.json
│   │   │   └── grading.json
│   │   └── without_skill/
│   │       ├── outputs/
│   │       ├── timing.json
│   │       └── grading.json
│   ├── eval-descriptive-name-2/
│   │   └── ...
│   └── benchmark.json
├── iteration-2/
│   └── ...
└── evals/
    └── evals.json
```

**Rules:**
- eval directories use **descriptive names** not numbers (e.g., `eval-multi-page-table-extraction`)
- Each iteration preserved in independent directory (do NOT overwrite previous)
- `_workspace/` not deleted — for post-verification and audit trail

## OpenCode-specific notes

When running skill tests inside an OpenCode harness session:

- Subagent spawning: use `task(subagent_type, prompt, run_in_background=true)` — default subagent type for generic work is `general`
- Timing capture: `<system-reminder>` notification contains `total_tokens` and `duration_ms` — capture in the orchestrator's first response after notification
- Skill loading: project-local `.opencode/skills/` is loaded automatically by OpenCode; no manual loading required
- Description trigger testing: harder to automate than Claude Code (no equivalent of `claude -p` headless trigger test); use the headless `opencode run --format json` probe pattern documented in this project's `AGENTS.md`
- Workspace location: place `_workspace/` inside the consumer project root (not in this plugin's repo)
