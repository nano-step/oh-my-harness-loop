## 1. Description writing patterns

The description is the only trigger mechanism for a skill. OpenCode looks at the available skills list and decides whether to use a skill based on `name` + `description` only.


### Understanding the trigger mechanism

The model tends NOT to call a skill for simple tasks it can handle with its base tools. A request like "read this PDF" may not trigger the skill even with a perfect description. The more complex, multi-step, and specialized the task, the higher the trigger probability.


### Writing principles

1. **What the skill does** + **specific trigger situations** — both must be stated
2. **Boundary conditions** — distinguish similar but non-trigger cases
3. **Be slightly "pushy"** — compensate for the model's conservative trigger tendency


### Good examples

```yaml
description: "Read, extract text/tables, merge, split, rotate, watermark,
  encrypt/decrypt, OCR, and perform all PDF operations. When .pdf files
  are mentioned or PDF output is requested, MUST use this skill. Especially
  useful when conversion/editing/analysis is needed — not just 'read this PDF'."
```

```yaml
description: "All spreadsheet work on .xlsx/.csv/.tsv files: add columns,
  formula calculation, formatting, charts, data cleaning. When user
  mentions a spreadsheet file — even casually ('the xlsx in Downloads') —
  use this skill."
```

### Bad examples

- `"A skill that processes data"` — too vague, unclear which file/task
- `"PDF-related work"` — no specific action list, no trigger situation stated

## 2. Body writing style

### Why-First principle

LLMs judge edge cases correctly when they understand the reasoning. Context delivery is more effective than coercive rules.


**Bad example:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good example:**
```markdown
Use pdfplumber for table extraction. PyPDF2 is specialized for text
extraction and does NOT preserve row/column structure of tables.
pdfplumber recognizes cell boundaries and returns structured data.
```

### Generalization principle

When problems are found from feedback or test results, generalize at the **principle level** instead of narrow fixes that only match a specific example.


**Overfit fix:**
```markdown
If a "Q4 sales" column exists, convert that column to numbers.
```

**Generalized fix:**
```markdown
If a column name contains keywords like "sales", "amount", "quantity"
that imply numeric values, convert that column to numeric type. If
conversion fails, keep the original value.
```

### Imperative tone

Use "do X" imperative form, not "you can..." or polite form. Skills are directives.


### Context conservation

Context window is a public good. Question whether every sentence justifies its token cost:
- "Is this something the model already knows?" → delete
- "Would the model make mistakes without this explanation?" → keep
- "Is one concrete example more effective than a long explanation?" → replace with example

## 3. Output format definition patterns

For skills where output format matters:

```markdown
## Report structure
Follow this template exactly:

# [Title]
## Summary
## Key findings
## Recommendations
```

Format definitions should be concise; including a real example is even more effective.

## 4. Example writing patterns

Examples are more effective than long explanations:

```markdown
## Commit message format

**Example 1:**
Input: Add JWT-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: Fix password-show button not working on login page
Output: fix(login): fix password show toggle button behavior
```

## 5. Progressive disclosure patterns

### Pattern 1: Domain separation

```
bigquery-skill/
├── SKILL.md (overview + domain selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)
```

When user asks about revenue, only `finance.md` is loaded.


### Pattern 2: Conditional detail

```markdown
# DOCX Processing

## Document generation
Generate new documents with docx-js. → see [DOCX-JS.md](references/docx-js.md).

## Document editing
Simple edits: modify XML directly.
**If tracked changes are needed**: see [REDLINING.md](references/redlining.md)
```

### Pattern 3: Large reference file structure

Reference files 300+ lines should include a table of contents at the top:

```markdown
# API Reference

## Table of contents
1. [Authentication](#authentication)
2. [Endpoint list](#endpoint-list)
3. [Error codes](#error-codes)
4. [Rate limits](#rate-limits)

---

## Authentication
...
```

## 6. Script bundling criteria

Observe transcripts of agents in test runs. The following patterns indicate bundling candidates:

| Signal | Action |
|--------|--------|
| 3 out of 3 tests generate the same helper script | Bundle into `scripts/` |
| Same `pip install` / `npm install` every time | Add dependency-install step to skill |
| Same multi-step approach repeated | Document as standard procedure in skill body |
| Similar error then same workaround every time | Document known issue + fix in skill |

Bundled scripts MUST go through execution testing.

## 7. Data schema standards

Use standard schemas for consistency across skill data exchange. Can be used for testing/evaluation of skills generated by the harness.

### eval_metadata.json

Per-test-case metadata:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "user's task prompt",
  "assertions": [
    "output contains X",
    "file was created in Y format"
  ]
}
```

### grading.json

Assertion-based grading results:

```json
{
  "expectations": [
    {
      "text": "output contains 'Seoul'",
      "passed": true,
      "evidence": "3rd stage: 'Seoul region data extraction' confirmed"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field name note:** use `text`, `passed`, `evidence` exactly (no `name` / `met` / `details` variants).


### timing.json

Execution time / token measurement:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Save `total_tokens` and `duration_ms` immediately upon subagent-completion notification. This data is only accessible at the notification moment and unrecoverable later.

## 8. What not to include in a skill

- README.md, CHANGELOG.md, INSTALLATION_GUIDE.md and other sidecar docs
- Skill-creation process meta info (test results, iteration history)
- User-facing manuals (skills are directives for AI agents)
- General knowledge the model already has

## 9. Skill reuse design

Before creating a new skill, check for duplication with existing skills. When building harnesses repeatedly, function-overlapping skills tend to accumulate under different names.

| Situation | Action |
|-----------|--------|
| Existing skill fully covers new function | Forbid new creation — link existing to the agent |
| Existing skill partially covers and can generalize | Generalize existing skill to extend |
| Intentional domain-specific partial overlap | Proceed with new creation — keep as separate skill |
| Function scope completely different | Proceed with new creation |

**Principle:** the more a skill focuses on a single role, the higher its reusability and the lower the duplication. If the role has 2+ aspects, first check if it can be split.


### How far to generalize

Generalization is infinitely possible, so stop at the **intended responsibility scope**. Keep intentional domain specificity, remove only accidental dependencies.

Example: a "fintech risk-assessment PDF" skill

| Stage | Result |
|-------|--------|
| Remove fintech dependency | "Assessment-result PDF" — stop here if scope is assessment report |
| Remove assessment dependency | "PDF formatting" — if it already exists, reuse it, don't create a new skill |

If the responsibility scope is intentionally specialized for "fintech risk assessment", do NOT generalize — keep as a separate skill.


Agents depending on the skill may change behavior. Check dependencies before extending; reflect the extended usage scope in the `description`.

## OpenCode-specific notes

When generating skills for OpenCode (not Claude Code):

- Skill location: `.opencode/skills/{name}/SKILL.md` (project) or `~/.config/opencode/skills/{name}/SKILL.md` (global)
- Both paths are auto-discovered by OpenCode
- Frontmatter matches OpenCode's expected schema (`name`, `description` are the trigger-relevant fields)
- For harness-team-generated skills, ship with the project (not the global config) by default
